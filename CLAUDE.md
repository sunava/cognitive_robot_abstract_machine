# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Overview

This is the **Cognitive Robot Abstract Machine (CRAM)** monorepo — a cognitive architecture for robot planning, perception, control, and knowledge representation. It is organized as a uv/Poetry workspace with the following workspace members (each under its own subdirectory with its own `pyproject.toml`):

| Package | Purpose |
|---|---|
| `pycram` | Cognitive planning, designators, plan graphs, ORM persistence |
| `giskardpy` | Whole-body motion control via constraint-based QP |
| `semantic_digital_twin` | World models, scene graphs, robot definitions |
| `krrood` | Knowledge representation: roles, ORM generation, EQL queries, RDR |
| `random_events` | Random variable/event definitions (C++ + Python hybrid) |
| `probabilistic_model` | Unified probabilistic model interface |
| `physics_simulators` | Simulation abstraction layer |
| `robokudo` | Perception pipeline for ROS2 |

All tests live centrally under `test/<package>_test/`.

## Installation

```bash
# Install uv (once)
curl -LsSf https://astral.sh/uv/install.sh | sh

# Install all workspace packages as editable installs
uv sync --active

# Alternative: Poetry
poetry install
```

## Architecture

### krrood — Knowledge Representation

The most architecturally novel package. Key subsystems:

- **Role Pattern** (`krrood/src/krrood/patterns/role/`): Runtime role assignment using `SubClassSafeGeneric[T]`. A `Role[T]` wraps a role-taker object, providing delegated attribute access. `role_transformer.py` uses `libcst` to generate stub files for role classes. Predicates (`HasRole[T]`, `HasRoleTaker`) enable type-safe role queries.

- **Ormatic** (`krrood/src/krrood/ormatic/`): ORM code-generation system using Jinja2 templates. Generates SQLAlchemy table classes from annotated Python dataclasses. Used by both `pycram` and `semantic_digital_twin` to persist plans and world state.

- **Entity Query Language (EQL)** (`krrood/src/krrood/entity_query_language/`): Pythonic relational query interface layered over SQLAlchemy. Allows filtering and joining entities with Python expressions.

- **Ripple Down Rules (RDR)** (`krrood/src/krrood/ripple_down_rules/`): Incremental knowledge acquisition system. Rule trees (single/multi-class) grow via exception cases without retraining. Supports serialisation and a user interface for interactive rule refinement.

- **Ontomatic** (`krrood/src/krrood/ontomatic/`): Tools for mapping ontology concepts to Python class hierarchies and property descriptors.

- **Singleton** (`krrood/src/krrood/singleton.py`): `SingletonMeta` metaclass with circular-dependency detection; instances can be cleared for test isolation.

### pycram — Planning

- **Fluent system** (`pycram/src/pycram/fluent.py`): Thread-safe proxy objects for reactive state. Support operator overloading, conditions, callbacks, and pulse semantics (NEVER/ONCE/ALWAYS).
- **Plan graph** (`pycram/src/pycram/plans/plan.py`): Tree-structured `rx.PyDiGraph` (rustworkx). Each `PlanNode` carries context, callbacks, and a snapshot of initial world state.
- **Designators** (`pycram/src/pycram/designators/`): Intent-based task specifications that are resolved lazily at execution time.
- **ORM** (`pycram/src/pycram/orm/`): Persists plan executions. Schema generated by krrood's Ormatic.

### semantic_digital_twin — World Model

- `world.py` (≈77 KB) is the central facade for the world state. All code interacting with robot environment goes through it.
- `robots/` contains per-robot definitions (PR2, HSR, TIAGo, Stretch, Tracy).
- `spatial_types/` provides `HomogeneousTransformationMatrix` and `Vector3` used throughout.
- `orm/` persists world state via krrood Ormatic-generated schemas.

### giskardpy — Motion Control

- Constraint-based whole-body control solved via QP.
- `motion_statechart/`: FSM-based motion composition.
- `executor.py`: Action execution engine, bridges planning intent to motion commands.
- Communicates with the rest of CRAM over ROS2 (Jazzy).

## Import Conventions

- **Within a package**: always use **absolute imports** (`from pycram.fluent import Fluent`).
- **In tests**: use **relative imports** for modules in the same test subfolder.
- **Types only**: guard with `TYPE_CHECKING` and import from `typing_extensions` (not `typing` or stdlib).

## Code Style

- Formatter: **Black** (PEP 8). Applied automatically by pre-commit hook.
- Type hints: use `typing_extensions` consistently.
- Methods should be short, low cyclomatic complexity, single-responsibility, no hidden side effects.
- 85% test coverage minimum per PR.

## Testing Notes

- Central fixtures (`test/conftest.py`) provide robot instances (PR2, HSR, TIAGo, Stretch, Tracy), semantic objects (Milk, Table, Apple), world, collision matrix, and krrood introspection tools.
- `krrood/pytest.ini` sets `log_cli = true` and `log_cli_level = DEBUG` for krrood tests.
- `random_events` C++ tests use Bazel (separate from pytest).
- Tests that need ROS2 require the ROS workspace to be built and sourced first.
